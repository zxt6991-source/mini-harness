# MiniHarness 容错与可靠性工程优化技术实现

## 1. 背景

本方案基于 `harness_engineering_guide/11_reliability` 的可观测性、反馈循环、容错模式和幻觉防护四类实践，结合当前 MiniHarness TypeScript 项目的已实现模块，定义一组可在现有架构内直接落地的可靠性优化。

指南中的核心目标可以归纳为：

- 可观测性：指标、结构化日志和 trace/span 贯穿一次运行。
- 容错：重试、超时、熔断、隔舱和错误作为观察，避免局部故障扩大。
- 反馈：高风险操作需要审批或人工介入，运行中需要安全点处理控制指令。
- 幻觉防护：工具调用前验证、输出交叉检查、置信度评估和人工审核。

当前项目已有基础：

| 能力 | 当前落点 | 状态 |
|---|---|---|
| 运行事件流 | `src/runtime/events.ts`、`Engine.runEvents()` | 已实现 |
| 工具并发隔舱 | `src/runtime/tool-scheduler.ts` 的 `maxConcurrentTools` | 已实现 |
| 工具超时 | `ToolScheduler` 和 `ToolExecutor` 的 Promise 级 timeout | 已实现 |
| 错误作为观察 | `toolErrorMode: observe` | 已实现 |
| 模型重试 | `src/runtime/retry.ts` + `Engine` | 已实现基础指数退避 |
| Provider 熔断 | `src/models/circuit-breaker.ts` + `ProviderRouter` | 已实现 |
| 输出治理与幻觉前置防护 | `src/models/output-governance.ts`、`src/tools/validation.ts` | 已实现 |
| 生产指标汇总 | `src/production/metrics.ts` | 已实现基础计数和平均延迟 |

主要缺口：

- 模型重试缺少 jitter，多个运行实例同时重试时容易形成同步重试峰值。
- 缺少通用 timeout 包装函数，模型或外部调用方无法复用统一超时错误语义。
- `tool_result` 事件只包含成功状态和延迟，不包含错误码、错误名、可恢复标记，不利于告警、恢复和问题聚类。
- `ProductionMetricsCollector` 缺少 P99、成功率、错误率和健康状态快照，不能直接支撑指南中的健康检查与告警阈值。
- 技术方案中的 `reliability/` 描述仍偏目标态，没有反映当前可靠性能力已经分散落在 `runtime`、`models`、`production` 和 `tools`。

## 2. 本次优化范围

本次采用渐进式优化，不引入外部监控后端，也不新增 UI/审批通道。改造目标是把现有可靠性底座补齐到可观测、可告警、可复用的程度。

### 2.1 重试与超时

修改 `src/runtime/retry.ts`：

- `RetryPolicyOptions` 增加 `jitterRatio?: number`。
- `defaultRetryPolicy` 默认 `jitterRatio: 0`，保持现有测试和行为稳定。
- `getRetryDelayMs()` 支持注入随机数函数，计算 `baseDelay + random(0, baseDelay * jitterRatio)`。
- 新增 `runWithTimeout()`，对任意异步操作施加 Promise 级超时，并支持可选 `createError` 生成框架内错误。

预期收益：

- 保持默认行为兼容。
- 生产配置可开启 jitter，降低雷鸣群式重试。
- 模型、工具、MCP 或业务扩展都能复用同一套 timeout 包装。

### 2.2 工具结果事件增强

修改 `src/runtime/events.ts` 和 `src/runtime/engine.ts`：

- `ToolResultEvent` 增加可选字段：`errorCode`、`errorName`、`retryable`。
- `Engine` 从 tool message metadata 中提取这些字段并写入 `tool_result`。
- `ToolScheduler` 的错误观察 metadata 增加 `retryable`，并对超时、临时模型/网络类错误保留可恢复线索。

预期收益：

- 指标 collector 可以按错误类型聚合工具失败。
- UI 或日志系统能显示明确失败原因。
- 后续可基于 `retryable` 做工具级重试或恢复建议。

### 2.3 生产指标健康快照

修改 `src/production/metrics.ts`：

- 记录工具延迟样本，输出 P50/P95/P99。
- 输出工具成功率、工具错误率、运行完成率。
- 支持 `ProductionMetricsCollectorOptions`：
  - `latencyWarningMs`
  - `errorRateWarningThreshold`
- `snapshot()` 增加 `health` 字段：
  - `status: 'healthy' | 'degraded' | 'critical'`
  - `alerts: Array<{ level; code; message; value; threshold }>`
- 统计 `toolErrors.byCode`，用于告警和事后分析。

预期收益：

- 对应指南中的健康检查与告警规则。
- 不绑定 Prometheus/DataDog 等外部后端，仍保持库级轻量。
- `main.ts` 可直接使用配置中的生产指标阈值。

### 2.4 配置接入

修改 `src/main.ts`：

- 创建 `ProductionMetricsCollector` 时传入：
  - `config.production.metrics.latencyWarningMs`
  - `config.production.metrics.errorRateWarningThreshold`

现有 `configs/harness.yaml` 和 `src/utils/config.ts` 已有这些配置字段，本次只完成实际接线。

## 3. 暂不纳入本次的能力

以下能力保留为后续阶段，原因是它们需要新的控制面、持久化或业务协议：

- 工具级自动重试：需要区分工具幂等性，避免重复写入、重复扣款、重复外部副作用。
- HITL 审批工作流：当前无用户审批 UI、通知通道和审批状态持久化。
- 运行时指令注入：需要独立控制通道和安全点队列。
- SAGA 补偿事务：需要业务工具声明补偿动作和幂等键。
- 输出事实核查：需要外部知识源、引用协议和人工审核策略。

## 4. 测试策略

采用现有 Vitest 测试风格，先写失败测试再实现。

新增或扩展测试：

- `tests/runtime-retry.test.ts`
  - 验证 jitter delay 计算。
  - 验证 `runWithTimeout()` 超时后抛出指定错误。
- `tests/runtime-events.test.ts`
  - 验证 `tool_result` 会携带失败工具的 `errorCode`、`errorName`、`retryable`。
- `tests/production-metrics.test.ts`
  - 验证健康快照包含 P99、成功率、错误率。
  - 验证高错误率或高延迟会产生 degraded/critical 告警。

最终验收命令：

```bash
pnpm test
pnpm typecheck
```

## 5. 与现有架构的关系

可靠性能力不再只理解为 `src/reliability/` 目录中的代码，而是一组横切能力：

```text
runtime/retry.ts             重试、jitter、通用 timeout
runtime/events.ts            traceId/sessionId 贯穿的运行事件协议
runtime/tool-scheduler.ts    工具隔舱、超时、错误作为观察
tools/executor.ts            工具权限、校验、超时、结构化日志
models/circuit-breaker.ts    Provider 熔断状态机
models/output-governance.ts  工具幻觉与注入防护
production/metrics.ts        运行指标、健康状态和告警快照
reliability/logger.ts        pino 结构化日志入口
```

这种布局保持了 MiniHarness 的轻量性：运行时负责产生事实事件，生产层负责聚合判断，模型层和工具层各自保留局部容错边界。

## 6. 验收标准

- 模型重试策略支持可配置 jitter，默认兼容旧行为。
- 通用 timeout 包装可以复用并返回框架内错误。
- 工具失败事件包含可聚合错误信息。
- 生产指标快照能直接回答成功率、错误率、P99 延迟和当前健康状态。
- `MiniHarness_TS_技术方案.md` 同步更新可靠性章节、生产指标章节、配置说明和阶段验收标准。
