# MiniHarness 评估与质量保障优化技术实现

## 1. 背景

本方案基于 `/Users/jojo/Desktop/all-agent/harness_engineering_guide/13_evaluation` 中的评估方法论、E2E 测试、基准测试、持续评估和 MiniHarness 测试策略，并结合当前 TypeScript 项目的实际架构制定。

第十三章的核心结论是：Agent Harness 的质量不能只看“有没有报错”，而要从三个层级量化：

- 步骤级：工具选择、参数、执行结果是否正确。
- 轨迹级：工具调用序列是否高效，是否存在重复调用，失败后能否恢复。
- 任务级：最终目标是否完成，以及耗时、token 和成本是否可接受。

当前 MiniHarness 已具备这些能力的基础：

| 能力 | 当前落点 | 状态 |
|---|---|---|
| 运行轨迹事件 | `src/runtime/events.ts`、`Engine.runEvents()` | 已实现 |
| 生产指标聚合 | `src/production/metrics.ts` | 已实现模型、工具、token、健康指标 |
| 输出治理 | `src/models/output-governance.ts` | 已实现工具调用前治理 |
| PGE 文本评估解析 | `src/orchestration/evaluator.ts` | 已实现 evaluator 输出标准化解析 |
| 单元/集成测试 | `tests/` | 已覆盖 runtime、tools、memory、models、mcp、production |

主要缺口：

- 缺少基于 `EngineEvent` 的离线评估器，无法把一次运行量化成步骤、轨迹、任务三层指标。
- `ProductionMetricsCollector` 缺少连续质量信号，例如任务成功率、平均每次运行 token、错误恢复率和重复工具调用率。
- 缺少基线回归检查 API，无法在 CI 或发布前判断质量是否退化。
- 主技术方案仍未把 `evaluation/` 作为独立子系统纳入总体架构。

## 2. 优化目标

1. 新增 `src/evaluation/` 子系统，从 `EngineEvent[]` 计算离线评估指标。
2. 提供聚合和回归检查 API，支撑测试集、E2E、benchmark 子集和发布前质量门禁。
3. 扩展 `ProductionMetricsCollector`，在原有健康快照之外输出连续评估质量信号。
4. 更新公共导出和主技术方案文档，使评估能力成为 MiniHarness 的一等模块。
5. 以 Vitest 覆盖新增能力，并保持现有 `pnpm test`、`pnpm typecheck`、`pnpm build` 可用。

## 3. 设计方案

### 3.1 `evaluation/` 子系统

新增 `src/evaluation/index.ts`，提供轻量、无外部依赖的评估 API：

```ts
evaluateEngineRun(events, expectations)
aggregateEvaluationResults(results)
checkEvaluationRegression(current, baseline, thresholds)
```

评估器只消费运行事件，不读取 `Engine` 内部状态，因此可以用于：

- 本地单元测试生成的 mock 轨迹。
- `Engine.runEvents()` 产出的真实轨迹。
- 后续从 JSONL、日志系统或 Langfuse 等平台导出的事件轨迹。

### 3.2 步骤级指标

步骤级指标来自 `model_message.toolCalls` 和 `tool_result`：

| 指标 | 来源 | 计算 |
|---|---|---|
| `toolAccuracy` | 实际 tool call 与 `expectedToolCalls` | 同位置工具名匹配数 / 期望工具数 |
| `parameterAccuracy` | 实际 tool call arguments 与期望 arguments | 同位置参数深度相等数 / 期望工具数 |
| `executionSuccessRate` | `tool_result.success` | 成功工具结果数 / 工具结果数 |

额外工具调用不直接降低步骤级准确率，而由轨迹级效率和重复率体现。这能区分“工具选择错了”和“走了弯路但最终恢复”。

### 3.3 轨迹级指标

轨迹级指标来自按时间排序的 `tool_result`：

| 指标 | 计算 |
|---|---|
| `trajectoryEfficiency` | `optimalToolNames.length / actualToolResults.length`，上限为 1 |
| `duplicateRate` | 连续相同工具名次数 / 实际工具调用数 |
| `errorRecoveryRate` | 失败工具调用后，下一个工具调用成功的比例 |
| `actualToolCalls` | 实际工具结果数量 |
| `optimalToolCalls` | 期望最优工具数量 |

该设计对应第十三章的轨迹长度效率、重复调用率和错误恢复率。

### 3.4 任务级指标

任务级指标来自 `agent_start`、`agent_end`、`runtime_error` 和 `model_message.usage`：

| 指标 | 计算 |
|---|---|
| `success` | 存在 `agent_end`，没有终止性 `runtime_error`，且最终内容包含期望文本 |
| `successRate` | 单次运行成功为 1，否则为 0；聚合时取平均 |
| `durationMs` | 最早事件到最晚事件的时间差 |
| `inputTokens/outputTokens/totalTokens` | 累加 `model_message.usage` |
| `costUsd` | 可选 `costPer1KTokens` 估算 |

当前实现先支持字符串包含式最终结果检查。复杂任务的语义评分可在后续通过独立 evaluator 或人工审核结果写入 `expectations`。

### 3.5 综合评分

单次运行的 `overallScore` 使用保守加权：

- 任务成功：40%
- 轨迹效率：30%
- 工具/参数准确：20%
- 成本效率：10%

如果未配置成本，成本维度按 1 处理；如果有成本，则以 `costReferenceUsd` 为参考上限做线性扣分。

### 3.6 基线回归检查

`checkEvaluationRegression()` 对聚合指标做阈值比较：

- 越高越好：`task.successRate`、`step.executionSuccessRate`、`trajectory.trajectoryEfficiency`、`overallScore`
- 越低越好：`task.avgDurationMs`、`task.avgCostUsd`、`trajectory.duplicateRate`

默认相对阈值：

- 允许质量指标下降 5%。
- 允许成本/耗时/重复率上升 10%。

返回结构化回归项，包含 metric、baseline、current、changeRatio 和 direction，可直接用于 CI 输出或发布门禁。

### 3.7 生产质量快照

扩展 `ProductionMetricsSnapshot`：

```ts
quality: {
  taskSuccessRate: number;
  averageRunDurationMs: number;
  averageTokensPerRun: number;
  errorRecoveryRate: number;
  duplicateToolCallRate: number;
}
```

这些字段来自运行时事件的连续聚合，不替代离线评估器。两者分工如下：

| 能力 | 使用场景 |
|---|---|
| `evaluation/` | 离线测试集、E2E、benchmark、发布门禁 |
| `production/metrics.ts` quality | 生产进程内实时质量概览和告警输入 |

## 5. 测试策略

采用当前项目的 Vitest 风格和 TDD 流程：

1. `tests/evaluation-metrics.test.ts`
   - 验证从事件轨迹计算步骤、轨迹、任务、token 和成本指标。
   - 验证重复调用、错误恢复和轨迹效率。
   - 验证聚合指标与基线回归检查。
2. `tests/production-metrics.test.ts`
   - 验证生产质量快照输出任务成功率、平均每次运行 token、错误恢复率和重复工具调用率。
3. 全量回归：

```bash
pnpm test
pnpm typecheck
pnpm build
```

## 6. 后续扩展

本轮不引入真实 LLM E2E、GAIA/WebArena/SWE-Bench 适配器或外部可观测性平台。后续可在当前 API 上继续扩展：

- 增加 JSONL 轨迹导入/导出。
- 增加 benchmark suite runner。
- 增加真实 LLM 慢速测试分组和环境变量门控。
- 增加基于独立 evaluator 的语义评分。
- 将质量快照导出到 Prometheus、OpenTelemetry 或 Langfuse。

## 7. 验收标准

- `evaluateEngineRun()` 能从 `EngineEvent[]` 生成三层评估指标。
- `aggregateEvaluationResults()` 能汇总多条运行结果。
- `checkEvaluationRegression()` 能识别成功率、耗时和综合分退化。
- `ProductionMetricsCollector.snapshot().quality` 能输出连续质量信号。
- 主技术方案文档同步体现 `evaluation/` 子系统和质量保障流程。
